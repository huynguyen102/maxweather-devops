The brief

"Max Weather" wants a weather-forecast platform on AWS. It is a platform, not an app — every choice serves four pillars distilled from the seven requirements.

Availability

Run 24/7, scale with traffic

Forecast traffic spikes in the morning — predictable and spiky. Autoscaling is core, and it must survive AZ/pod failure.

API + Security

APIs, protected by OAuth2

The product is the API that Max Weather's own frontend developers consume — with a contract and access control.

Delivery

CI/CD, reproducible IaC

Ship safely (staging → prod) and rebuild anywhere from parameterized Terraform — never by hand.

Observability

Centralized logs

Application logs flow to CloudWatch — the eyes needed to operate the platform.

Architecture

The request path (data plane) and what manages it (control plane).

Frontend dev │ HTTPS + Bearer token ▼ API Gateway ──▶ Lambda Authorizer ──▶ Cognito (JWKS) │ proxy ▼ NLB ─▶ Nginx Ingress ─▶ Service ─▶ Pods (Flask) ─▶ Open-Meteo │ stdout ─▶ Fluent Bit ─▶ CloudWatch

Control plane vs data plane

Data plane carries user traffic — every hop above is on the critical path. Control plane (EKS API/scheduler, HPA, Cluster Autoscaler, Cognito, Jenkins, Terraform) decides and manages but sits off the path. If the control plane degrades, running pods keep serving; you only lose the ability to change. That split drives incident triage.

Why autoscaling is core

Spiky, predictable load forces the choice: fixed capacity either wastes money off-peak or falls over at peak. Autoscaling is the only way to be both available and cost-efficient — scale out protects availability, scale in controls cost, no manual toil. Two tiers: HPA scales pods on CPU; Cluster Autoscaler scales nodes when pods can't fit.

Security & logging

OAuth2 via Cognito (client_credentials) as issuer; a custom API Gateway Lambda authorizer validates the JWT (signature via JWKS, issuer, expiry, scope). Pods run in private subnets, egress via NAT. App writes structured JSON to stdout; the CloudWatch observability addon (Fluent Bit) ships it — the app holds no AWS credentials, so it stays portable.

What was proven

  • API + OAuth2 end to end — token issued, authorized call returns forecast (200), no-token call blocked (401): API Gateway → Lambda authorizer → NLB → Ingress → pod → Open-Meteo.
  • Autoscaling under real load — load on /health drove CPU past target; HPA scaled 2→10 in ~70s, then back down.
  • Logs in CloudWatch — structured app logs ("message":"forecast served") enriched with pod/namespace/image metadata by Fluent Bit.

Lessons from the live run

Five real issues that validate could not catch — only deploying could. Each is fixed and documented.

  • Arch mismatch — an image built on Apple Silicon (arm64) won't run on x86 EKS nodes ("no match for platform"). Fix: docker build --platform linux/amd64.
  • Ingress collision — two catch-all Ingresses can't coexist on one nginx controller. Fix: staging gets its own host; prod keeps the catch-all API Gateway proxies to.
  • API Gateway needs a real backend — the integration can't be created with an empty URL, and the NLB doesn't exist until k8s runs. Hence the two-phase apply.
  • Teardown order — delete the k8s LoadBalancer (NLB) and ECR images before terraform destroy, or VPC/repo deletion fails.
  • Docker-out-of-Docker — a CI smoke test can't curl localhost a published port; test inside the container instead.

Provision guide

Stand the whole thing up, run it, and tear it down. Region ap-southeast-1. Full commands in jenkins/RUNBOOK.md and k8s/platform/README.md.

Costs money. This provisions EKS + NAT + NLB. Tear down when done (step 6).
1 Infrastructure (Terraform)

Two-phase: the API Gateway is wired to the ingress NLB only after the cluster exists.

cd terraform
terraform init
terraform apply                 # ~15-20 min (EKS)

Outputs give you the cluster name, ECR URL, Cognito endpoints, API endpoint.

2 Platform (once per cluster)

Cluster-wide pieces the app depends on — not in the app pipeline.

aws eks update-kubeconfig --name "$(terraform output -raw cluster_name)" --region ap-southeast-1
kubectl apply -f ../k8s/platform/namespaces.yaml
kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/download/v0.7.2/components.yaml
kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/controller-v1.11.3/deploy/static/provider/aws/deploy.yaml
# cluster-autoscaler: substitute role ARN + cluster name (see k8s/platform/README.md)

metrics-server feeds the HPA · ingress-nginx creates the NLB · cluster-autoscaler scales nodes.

3 Wire API Gateway to the NLB (phase-2 apply) 
NLB=$(kubectl -n ingress-nginx get svc ingress-nginx-controller \
        -o jsonpath='{.status.loadBalancer.ingress[0].hostname}')
terraform apply -var api_backend_url="http://$NLB"
4 Deploy the app — via Jenkins

Start the tool-equipped Jenkins host, then run the pipeline.

cd ../jenkins
docker compose up -d --build     # Jenkins + docker/aws/kubectl/kustomize
# open http://localhost:8080 (no login)
# New Item -> Pipeline -> "Pipeline script from SCM"
#   Git URL: file:///workspace/maxweather   Script: jenkins/Jenkinsfile
# Build with Parameters (ECR_REPO, CLUSTER_NAME, AWS_REGION) -> approve prod

The pipeline builds the image, pushes to ECR, deploys to staging, waits for approval, then promotes the same image to prod.

4b Deploy the app — by hand (alternative) 
REPO=$(terraform output -raw ecr_repository_url); SHA=$(git rev-parse --short HEAD)
aws ecr get-login-password --region ap-southeast-1 | docker login --username AWS --password-stdin "$REPO"
docker build --platform linux/amd64 -t "$REPO:$SHA" ../app
docker push "$REPO:$SHA"
( cd ../k8s/overlays/prod && kustomize edit set image maxweather-app="$REPO:$SHA" )
kubectl apply -k ../k8s/overlays/prod
kubectl -n prod rollout status deploy/maxweather
5 Verify (Postman / newman) 
cd ../terraform
newman run ../postman/MaxWeather.postman_collection.json \
  --env-var base_url="$(terraform output -raw api_endpoint)" \
  --env-var token_url="$(terraform output -raw cognito_token_endpoint)" \
  --env-var client_id="$(terraform output -raw cognito_client_id)" \
  --env-var client_secret="$(terraform output -raw cognito_client_secret)" \
  --env-var scope="$(terraform output -raw cognito_scope)"

All three requests pass — including the negative test that a missing token is rejected.

6 Tear down (order matters) 
kubectl delete svc ingress-nginx-controller -n ingress-nginx   # removes the NLB first
# delete ECR images (repo must be empty), then:
cd terraform && terraform destroy
cd ../jenkins && docker compose down
