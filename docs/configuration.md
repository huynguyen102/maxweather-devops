# Configuration Reference

The full input/output contract for the Terraform. Each module also carries a `README.md` generated by `terraform-docs` from its own `variables.tf`/`outputs.tf`; this page aggregates them and explains how the pieces are parameterized and wired.

- Naming/tagging behaviour: [naming-and-tagging.md](naming-and-tagging.md)
- Architecture: [architecture.md](architecture.md)

## How it is wired
The root (`terraform/`) owns the provider, the naming prefix, and the identity tags, then calls each module and threads outputs between them:

```
vpc ──subnets──▶ eks ──cluster_name, node_role_name──▶ cloudwatch
                  │
ecr               └──(image pulled by app on EKS)
cognito ──issuer, jwks_uri, client_id, scope──▶ lambda-authorizer ──invoke_arn, function_name──▶ api-gateway
```

Everything environment-specific is a variable with a default; `terraform.tfvars` supplies the values for this deployment. This is what makes the stack reproducible in another environment or region (requirement #7).

## Running it
```sh
cd terraform
terraform init
terraform validate                       # no credentials needed
terraform plan  -var-file=terraform.tfvars   # needs AWS credentials
terraform apply -var-file=terraform.tfvars
```
`terraform.tfvars` is loaded automatically; the `-var-file` flag is shown for clarity.

## Root inputs (`terraform/variables.tf`)

| Name | Type | Default | Description |
|---|---|---|---|
| `project` | string | `maxweather` | First segment of every name; `Project` tag. |
| `environment` | string | _(required)_ | Infra tier (e.g. `prod`); second name segment. App staging/prod are k8s namespaces. |
| `region` | string | _(required)_ | AWS region to deploy into. |
| `vpc_cidr` | string | `10.0.0.0/16` | CIDR block for the VPC. |
| `az_count` | number | `2` | Number of AZs to spread the network across (≥2 for HA). |
| `single_nat_gateway` | bool | `true` | One NAT (cheaper) vs one per AZ (HA egress). |
| `cluster_version` | string | `1.31` | Kubernetes version for the control plane. |
| `node_instance_types` | list(string) | `["t3.medium"]` | Instance types for the node group. |
| `node_desired_size` | number | `2` | Desired node count. |
| `node_min_size` | number | `2` | Minimum nodes (autoscaler floor). |
| `node_max_size` | number | `4` | Maximum nodes (autoscaler ceiling). |
| `eks_endpoint_public_access` | bool | `true` | Expose the Kubernetes API endpoint publicly. |
| `eks_public_access_cidrs` | list(string) | `["0.0.0.0/0"]` | CIDRs allowed to reach the public API endpoint. |
| `api_backend_url` | string | `""` | URL the API Gateway proxies to (Nginx Ingress NLB). Set at phase 8. |
| `log_retention_days` | number | `14` | Retention for Container Insights log groups. |

## Parameterizing for another environment or region
Because every environment-specific value is a variable, standing the stack up elsewhere is a matter of values, not code:

```sh
# a second region (e.g. DR — see ADR-0006), same modules, different values
terraform apply -var region=ap-southeast-2 -var environment=prod -var vpc_cidr=10.1.0.0/16
```

Or keep separate value files (e.g. `prod.ap-southeast-1.tfvars`) and select one at apply time. No module code changes.

## Two-phase apply (Terraform ↔ Kubernetes seam)
`api_backend_url` is empty on the first apply because the backend it points to — the Nginx Ingress NLB — does not exist until the Kubernetes manifests are applied to the running cluster. The sequence at deploy time (phase 8) is:

1. `terraform apply` — provisions VPC, EKS, ECR, Cognito, authorizer, API Gateway, CloudWatch.
2. `kubectl apply` the Kubernetes artifacts — the Nginx Ingress Controller provisions an NLB and gets a public hostname.
3. `terraform apply -var api_backend_url=http://<nlb-hostname>` — points the API Gateway proxy at the ingress.

This is a genuine boundary between the infrastructure plane (Terraform) and the workload plane (Kubernetes), not a workaround.
